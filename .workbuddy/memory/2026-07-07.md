## 页面乱码修复 — MySQL字符集 + Nginx charset
- 用户报告页面中文数据显示乱码（静态UI正常，API返回的数据乱码）
- 排查发现 **MySQL客户端字符集为 latin1**（非utf8mb4），导致之前用SQL脚本插入种子数据时UTF-8中文被双重编码（mojibake）
  - 正确的"花花"HEX应为 `E88AB1E88AB1`，实际存储为 `C3A8C5A0C2B1C3A8C5A0C2B1`（每个字节被当作Latin-1再编码为UTF-8）
- **修复数据**：在SQL脚本头部添加 `SET NAMES utf8mb4;`，DROP所有表后重建并重新插入种子数据，使用 `--default-character-set=utf8mb4` 执行
- **永久修复**：在MySQL容器 `/etc/mysql/conf.d/charset.cnf` 写入 `[client] default-character-set=utf8mb4` 配置，重启MySQL容器
- 验证：character_set_client/connection/results 全部从 latin1 → utf8mb4，API返回中文数据正确
- 同时修复 Nginx 响应头：host-nginx 和 frontend nginx.conf 添加 `charset utf-8; charset_types text/css text/javascript application/javascript application/json;`
- 修复后所有资源 Content-Type 均带 `charset=utf-8`
- Feeders API 返回 realName/serviceArea 为 N/A，可能字段映射问题（后续排查）
- Dashboard stats 返回 None，可能字段名不匹配（后续排查）
